# 2. 每日世界币发放 & 回收

**触发**：每日 UTC0-12:00（北京时间 20:00 / 美东时间 08:00）定时任务，按步骤顺序执行。

```
每日 UTC0-12:00 触发
  │
  ├─ 【步骤 1：用户每日数据快照】
  │     │
  │     └─ 遍历所有注册用户，写入快照（世界币余额、排名、合约交易额）
  │
  ├─ 【步骤 2：回收昨日赠送】
  │     │
  │     ├─ 取昨日赠送金额
  │     ├─ 昨日有世界币支出（竞猜投入 / 抽奖消耗）？
  │     │     ├─ 是 → 跳过，不回收
  │     │     └─ 否 ↓
  │     ├─ 回收金额 = min（昨日赠送 ÷ 2，当前余额）
  │     ├─ 写负数流水（类型：每日赠送回收，金额 = 回收金额）
  │     └─ 发送站内信（通知回收金额）
  │
  └─ 【步骤 3：发放今日赠送】
        │
        ├─ 取今日赠送数量
        ├─ 遍历所有注册用户，各写一条正数流水（类型：每日赠送）
        ├─ 发送站内信（通知发放金额）
        └─ 有绑定邮箱？是 → 发送邮件；否 → 跳过
```

## 2.1 用户每日数据快照

**用途**：供落地页 Banner 区展示用户数据与昨日的变化量，须在回收 & 发放前执行，确保捕获昨日自然日结束时的状态。

1. 遍历所有注册用户
1. 取当前世界币余额（世界币流水表最新期末余额）
1. 取最近一次排行榜清算写入的预估排名
1. 取用户活动期累计合约交易额
1. 写入用户每日快照表，快照日期 = 当前活动日（美东时间）

## 2.2 回收昨日赠送

1. 取昨日每日配置表的赠送世界币数量作为赠送金额
1. 检查昨日是否存在世界币支出流水（类型：竞猜投入 / 抽奖消耗），若有则跳过，不回收
1. 回收金额 = min（赠送金额 ÷ 2，当前余额），回收额不超过当前余额
1. 写一条负数世界币流水（类型：每日赠送回收，金额 = 回收金额）
1. 向该用户发送站内信告知回收金额
   - 标题：世界币回收通知
   - 正文：您昨日共获得赠送世界币 {赠送金额} 个，因未使用本次扣除 {回收金额} 个。

## 2.3 发放今日赠送

1. 取今日每日配置表的赠送世界币数量
1. 给所有注册用户各写一条正数世界币流水（类型：每日赠送）
1. 向每位用户发送站内信告知发放金额；若用户已绑定邮箱，同时发送邮件
   - 标题：世界币每日赠送
   - 正文：您今日获得赠送世界币 {赠送金额} 个，请在今日内用于竞猜或抽奖，未使用部分次日将回收一半。

## 2.4 注册即时赠送

**触发**：用户注册成功，事件驱动。

**背景**：每日批量赠送在 UTC 12:00 对已注册用户执行；当日批量赠送完成后注册的用户无法等到次日才获得第一笔世界币，需在报名时立即补发当日赠送。

```
用户注册成功
  │
  ├─ 活动进行中？
  │     ├─ 否 → 跳过
  │     └─ 是 ↓
  │
  ├─ 今日批量赠送（步骤 3）已完成？
  │     ├─ 否 → 跳过（批量任务届时会覆盖）
  │     └─ 是 ↓
  │
  ├─ 取今日赠送数量（每日配置表）
  ├─ 写正数世界币流水（类型：每日赠送）
  └─ 发送站内信（通知发放金额）
```

1. 判断活动状态 = 进行中，否则跳过
2. 判断今日批量赠送（2.3）是否已完成，未完成则跳过（批量任务会在当日 UTC 12:00 统一发放）
3. 取今日每日配置表的赠送世界币数量
4. 写一条正数世界币流水（类型：每日赠送）
5. 向用户发送站内信告知发放金额

# 3. 竞猜

## 3.1 赛事状态机

```
[赛程同步]
  │ 触发：自动每日 UTC 06:00 / 手动后台「同步赛程」按钮
  │ 同步：赛事ID、主客队名称、开赛时间、赛段（stage）、分组（group）（来自 football-data.org）
  │ 写入：新赛程写入，状态 = 待配置；已存在则更新队名和开赛时间，不覆盖赔率
  ▼
待配置
  │ [运营后台完成配置]
  │   1. 选择配置日期
  │   2. 从已同步赛程中选择当日赛事（最多 2 场）
  │   3. 填写主胜 / 平局 / 客胜赔率
  │   4. 填写 X 帖子链接、每日赠送数量、抽奖配置
  │   5. 保存 → 写入每日配置表（是否已生效 = 否）
  ▼
已配置
  │ [每日 UTC0-12:00，是否已生效 → 是]
  ▼
待开赛
  │ [定时任务：到达开赛时间]
  │   更新赛程表状态 = 进行中
  │   前端：竞猜按钮置灰或隐藏
  │   后端：赛事状态 ≠ 待开赛时拒绝下注（防止绕过前端直接调接口）
  ▼
进行中（竞猜截止）
  │ [开赛 85 分钟后每 15 分钟轮询]
  ├─ 未结束 → 继续轮询
  │
  ▼
已结束 → 触发竞猜结算
```

## 3.2 竞猜记录状态机

```
[用户下注] ──→ 待结算
                  │
                  ├─ [竞猜方向 = 赛果] ──→ 胜（派发世界币）
                  └─ [竞猜方向 ≠ 赛果] ──→ 负
```

## 3.3 下注扣币

**触发**：用户提交竞猜，实时执行。

```
用户提交竞猜
  │
  ├─ 赛事状态 = 待开赛？
  │     ├─ 否 → 拒绝，返回提示：竞猜已截止
  │     └─ 是 ↓
  │
  ├─ 投入世界币 ≥ 当日赠送数量？
  │     ├─ 否 → 拒绝，返回提示：今日竞猜金额不得小于 {当日赠送世界币数量}
  │     └─ 是 ↓
  │
  ├─ 余额 ≥ 投入世界币？
  │     ├─ 否 → 拒绝（余额不足）
  │     └─ 是 ↓
  │
  ├─ 写负数世界币流水（类型：竞猜投入）
  └─ 写竞猜记录（结算结果留空，状态：待结算）
```

1. 校验赛事状态 = 待开赛，否则拒绝，返回提示「竞猜已截止」
2. 校验投入世界币数量 ≥ 当日每日配置表的赠送世界币数量，不足则拒绝，返回提示「今日竞猜金额不得小于 {当日赠送世界币数量}」
3. 校验用户当前余额 ≥ 投入世界币数量，不足则拒绝
4. 写一条负数世界币流水（类型：竞猜投入）
5. 写一条竞猜记录，结算结果留空，状态待结算

## 3.4 结算

**触发**：开赛 85 分钟后开始每 15 分钟轮询 football-data.org，检测到赛事状态变为已结束时执行。

```
赛事状态 → 已结束
  │
  ├─ 读取赛程表赛果
  │
  └─ 遍历该赛事所有竞猜记录
        │
        ├─ 竞猜方向 = 赛果？
        │     │
        │     ├─ 是（胜）→ 派发世界币 = 投入 × 赔率快照
        │     │              写正数流水（类型：竞猜胜出）
        │     │              更新竞猜记录（结果：胜、派发世界币、结算时间）
        │     │
        │     └─ 否（负）→ 更新竞猜记录（结果：负、结算时间）
        │                    不写世界币流水
        │
        └─ 继续下一条，直至遍历完毕
```

1. 读取赛程表的赛果
2. 遍历该赛事所有竞猜记录
3. 若竞猜方向 = 赛果（胜）：
   - 派发世界币 = 投入世界币 × 赔率快照
   - 写一条正数世界币流水（类型：竞猜胜出）
   - 更新竞猜记录：结算结果 = 胜、派发世界币、结算时间
4. 若竞猜方向 ≠ 赛果（负）：
   - 更新竞猜记录：结算结果 = 负、结算时间
   - 不写世界币流水（世界币已在下注时扣除）

# 4. 任务奖励

## 4.1 连续签到奖励

**触发**：用户点击签到，实时执行。

```
用户点击签到
  │
  ├─ 活动进行中？
  │     ├─ 否 → 拒绝（活动已结束，签到入口关闭）
  │     └─ 是 ↓
  │
  ├─ 当日（美东时间）已签到？
  │     ├─ 是 → 拒绝（今日已签到）
  │     └─ 否 ↓
  │
  ├─ 写签到记录（日期、当前连签天数）
  ├─ 写正数世界币流水（类型：连续签到，金额 = 本天对应奖励）
  ├─ 发送站内信及邮件通知用户发放金额
  └─ 返回成功，前端更新签到状态
```

1. 校验活动状态 = 进行中，否则拒绝，签到入口同步关闭
2. 校验用户当日（美东时间）无签到记录，否则拒绝（幂等）
3. 取用户当前连签天数，计算本次为第几天（断签则重置为第 1 天）
4. 写一条签到记录（含日期、连签天数）
5. 写一条正数世界币流水（类型：连续签到，金额 = 签到天数对应奖励）
6. 向用户发送站内信；若已绑定邮箱，同时发送邮件
7. 返回成功响应，前端收到后更新签到格样式并展示发放成功弹窗

## 4.2 社交任务奖励

**触发**：用户点击「我已关注 / 我已评论 / 我已加入」确认按钮，实时执行。无需第三方验证，用户自行确认即视为完成。

```
用户点击确认按钮
  │
  ├─ 活动进行中？
  │     ├─ 否 → 拒绝
  │     └─ 是 ↓
  │
  ├─ 幂等校验
  │     ├─ 一次性任务（关注X / 关注Telegram）：该用户该任务是否已有完成记录？
  │     │     ├─ 是 → 拒绝（已完成）
  │     │     └─ 否 ↓
  │     └─ 每日任务（评论X帖子）：该用户今日（美东时间）该任务是否已完成？
  │           ├─ 是 → 拒绝（今日已完成）
  │           └─ 否 ↓
  │
  ├─ 写社交任务记录（任务类型、任务日期、状态：已完成、完成时间）
  ├─ 写正数世界币流水（类型：社交任务，金额：+10）
  └─ 返回成功，前端更新按钮为已完成置灰状态，展示发放成功弹窗
```

1. 校验活动状态 = 进行中，否则拒绝
2. 幂等校验：
   - 一次性任务（关注X / 关注Telegram）：全局唯一，有完成记录则拒绝
   - 每日任务（评论X帖子）：当日（美东时间）唯一，今日已完成则拒绝
3. 写一条社交任务记录（任务类型、任务日期、状态：已完成、完成时间）
4. 写一条正数世界币流水（类型：社交任务，金额：+10）
5. 返回成功响应，前端更新按钮样式并展示发放成功弹窗

## 4.3 合约交易梯度奖励

**触发**：合约业务系统每笔成交后通过消息通知增长系统，实时执行。

```
用户合约成交
  │
  ├─ 累加用户当日有效合约交易额
  │
  └─ 遍历合约任务配置表（档位序号升序）
        │
        ├─ 当日累计交易额 ≥ 本档门槛？
        │     ├─ 否 → 停止遍历
        │     └─ 是 ↓
        │
        ├─ 本档今日已领取？
        │     ├─ 是 → 检查下一档
        │     └─ 否 ↓
        │
        ├─ 写正数世界币流水（类型：合约任务，金额 = 本档奖励）
        ├─ 写合约任务领取记录（含交易额快照）
        └─ 检查下一档
```

1. 累加用户当日合约交易额（所有成交均计入，不做风控过滤）
2. 遍历合约任务配置表各档位，按档位序号升序检查
3. 若当日累计交易额 ≥ 本档门槛，且合约任务领取记录中该档今日未领取：
   - 写一条正数世界币流水（类型：合约任务，金额 = 本档奖励世界币数量）
   - 写一条合约任务领取记录（含当日累计交易额快照）
4. 继续检查下一档，直到未达标为止

# 5. 抽奖结算

**触发**：用户触发抽奖，实时执行。

```
用户触发抽奖
  │
  ├─ 当日已抽次数 < 每日次数上限？
  │     ├─ 否 → 拒绝（次数已达上限）
  │     └─ 是 ↓
  │
  ├─ 余额 ≥ 抽奖消耗世界币？
  │     ├─ 否 → 拒绝（余额不足）
  │     └─ 是 ↓
  │
  ├─ 写负数世界币流水（类型：抽奖消耗）
  │
  ├─ 按权重随机抽取奖品（同时检查当日剩余数量上限）
  │
  ├─ 写抽奖记录，当日已抽次数 +1
  │
  └─ 中奖？
        ├─ 是 → 写奖励记录（来源：抽奖，状态：待审核）
        └─ 否 → 结束
```

1. 校验用户当日已抽次数 < 每日配置表的每日抽奖次数上限，超限则拒绝
2. 校验用户当前余额 ≥ 每日配置表的抽奖消耗世界币，不足则拒绝
3. 写一条负数世界币流水（类型：抽奖消耗）
4. 按抽奖奖品配置表的中奖权重随机抽取奖品
5. 写一条抽奖记录，更新当日已抽次数
6. 若中奖，写一条奖励记录（来源：抽奖，状态：待审核）

# 6. 排行榜清算

## 6.1 定时清算

**触发**：每 15 分钟一次。

**说明**：定时清算仅更新预估排名和预估奖励，供落地页实时展示，不生成正式奖励记录。

### 清算步骤

```
每 15 分钟触发
  │
  ├─ 1. 创建排行榜清算日志（状态 = 进行中，记录 range_start / range_end）
  │
  ├─ 2. 读取世界币流水表，取所有用户的当前世界币余额
  │
  ├─ 3. 计算排名
  │     └─ 按世界币余额降序排列
  │        并列处理：余额相同时，按注册时间升序（先注册者排名靠前）
  │
  ├─ 4. 更新预估奖励
  │     └─ 按排名对照奖品表，将当前排名对应奖励写入排行榜快照
  │        （供落地页展示，不生成奖励记录）
  │
  └─ 5. 更新清算日志（状态 = 已完成，记录完成时间）
```

### range_start / range_end 取值规则

| 情况 | range_start | range_end |
|---|---|---|
| 首次清算 | 活动开始时间 | 本次触发时间 |
| 非首次清算 | 上次清算的 range_end | 本次触发时间 |
| 活动结束时间 ≤ 本次触发时间 | 上次清算的 range_end | 活动结束时间，本次即为最终清算 |

### 串行约束

同一活动的定时清算串行执行：若上次清算尚未完成，等待其完成后再执行本次。

## 6.2 最终清算

**触发**：活动结束后，6.1 的最后一次执行即为最终清算（range_end = 活动结束时间）。

**说明**：最终清算在完成 6.1 的全部步骤后，额外执行以下操作：

```
6.1 最后一次执行完成（range_end = 活动结束时间）
  │
  ├─ 以当轮排名结果为准，对照奖品表确认各用户最终奖励
  │
  └─ 遍历所有最终奖励 > 0 的用户
        └─ 写奖励记录（来源：排行榜，状态：待审核）
```

1. 以当轮排名结果为准，按排名对照奖品表确认每位上榜用户的最终奖励金额
2. 为所有最终奖励 > 0 的用户各写一条奖励记录（来源：排行榜，状态：待审核）
3. 不触发通知，通知由奖励发放流程处理

# 8. 边界场景

| 编号 | 场景 | 期望行为 |
|---|---|---|
| B-1 | 上次定时清算未完成，下次触发时间已到 | 等待上次完成后再执行，不并发 |
| B-2 | 活动结束时间不在整双数点 | 下一个整双数点触发时，range_end 取活动结束时间，直接作为最终清算执行 |
| B-3 | 活动被提前终止 | 触发一次终止清算（视为最终清算），之后不再排定新的清算 |
| B-5 | 竞猜赛事被取消或延期 | 待澄清：是否退还投入世界币 |
| B-6 | 世界币余额为 0，触发每日赠送回收 | 跳过，不写回收流水 |
